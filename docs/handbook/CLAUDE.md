# AI Handbook

## Project Purpose

Company AI usage handbook — two living documents for learning, sharing, and training:
- `handbook-general.md` — for everyone in the company (non-technical AI usage)
- `handbook-dev.md` — for the engineering team (AI in development workflows)

## Workflow

The user will provide raw content in any form — AI conversation replies, chat logs, copy-pasted articles, rough notes, or anything else. The content is raw material, not final text.

1. **Extract** — identify the actionable insights, useful patterns, and key takeaways from the raw content. Ignore filler, pleasantries, repetition, and generic advice.
2. **Plan first** — propose which handbook, which section, and a brief outline of what you'd write. Wait for approval.
3. **Copywrite** — after approval, rewrite the extracted gems into **Dos & Don'ts** format (table with Do / Don't columns) under the relevant section
4. **Place** — add to the correct handbook (general vs tech) and section
5. **Create sections** as needed — don't force content into wrong categories

## Rules

1. **Don't invent policies** — only document what the user states. Ask for clarification on sensitive topics (data handling, IP, compliance).
2. **Practical over theoretical** — every concept needs a concrete example or workflow.
3. **Opinionated** — state what we recommend and why. No "it depends" without guidance.
4. **Audience-aware** — write for the team's actual skill level. New hires should be able to follow.
5. **No fluff** — every section should teach something or enable an action.
6. **Security by default** — never include real API keys, tokens, or internal URLs. Use placeholders.
7. **Version awareness** — note tool/API versions and dates. These change fast.

## Writing Style

- Clear, direct prose. Short paragraphs.
- Second person ("you") for instructions, first plural ("we") for team decisions.
- Code examples must be runnable — no pseudocode unless labeled.
- Include "Why" sections — people follow rules better when they understand the reasoning.
